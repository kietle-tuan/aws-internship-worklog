# WORKSHOP

## AWS Stock Analyzer Project Testing Workshop

---

## 1. Overview

This workshop describes the practical testing activities related to the final project **AWS Stock Analyzer**.

In this project, I participated as a **QA Tester**. My main responsibility was to run or access the project, check the application behavior, test basic functions, and record testing observations.

My role was not to independently deploy the whole AWS infrastructure. Instead, I focused on checking whether the application worked correctly from the user and tester perspective.

---

| Item | Description |
|---|---|
| Project Name | AWS Stock Analyzer |
| Internship Topic | AWS Cloud Computing |

| Testing Level | Basic manual testing |

---

## 3. Workshop Objectives

The main objectives of this workshop were:

- Understand the main functions of the AWS Stock Analyzer project.
- Run or access the project based on the team setup.
- Check whether the frontend interface displayed correctly.
- Test stock symbol input cases.
- Observe backend response behavior.
- Check whether frontend and backend worked together.
- Record testing observations and provide feedback to the team.

The project was tested from the user and QA Tester perspective.

| Testing Item | Description |

---

## 5. Workshop Process

### Step 1: Review Project Functions

Before testing, I reviewed the main purpose and expected functions of the **AWS Stock Analyzer** project.

The main function of the application was to allow users to enter a stock symbol and receive stock-related analysis information.

This step helped me understand the expected behavior before starting the testing process.

---

### Step 2: Run or Access the Application

After reviewing the project, I ran or accessed the application based on the team’s setup.

At this step, I checked whether:

- The application could open normally.
- The main page was displayed.
- The input field was visible.
- The search or submit button was available.
- The result display area was shown properly.

This helped confirm that the project was ready for basic manual testing.

---

### Step 3: Test Frontend Interface

I manually checked the frontend interface of the application.

The checked items included:

- Main page layout
- Text readability
- Stock symbol input field
- Button display and response
- Result display section
- Basic user interaction flow

The purpose was to check whether the interface was understandable and usable from the user perspective.

---

### Step 4: Test Input Cases

I tested several basic input cases to observe how the application handled user input.

| Test Case | Expected Result | Status |
|---|---|---|
| Valid stock symbol | The system processes the request and displays stock-related result | Checked |
| Empty input | The system should not process invalid input silently | Checked |
| Invalid stock symbol | The system should handle the error properly | Checked |
| Repeated search | The system should continue responding normally | Checked |

This step helped me check basic input validation and user flow behavior.

---

### Step 5: Check Backend Response

I observed backend response behavior at a basic level.

The checked points included:

- Whether the backend received requests from the frontend.
- Whether the backend returned a response.
- Whether the response could be displayed on the frontend.
- Whether invalid input cases were handled properly.
- Whether the system behavior matched the expected user flow.

This was manual checking and observation, not advanced automated API testing.

---

### Step 6: Check Frontend and Backend Integration

After checking the frontend and backend behavior, I tested the basic integration flow.

The main flow was:

1. The user enters a stock symbol.
2. The frontend sends the request.
3. The backend processes the request.
4. The backend returns the result.
5. The frontend displays the result to the user.

This step helped confirm whether the main application flow worked from input to result display.

---

### Step 7: Record Testing Observations

After testing, I recorded testing observations and shared feedback with the team.

The recorded information included:

- Passed cases
- Failed cases if found
- UI issues
- Input validation issues
- Backend response issues
- Integration issues
- Suggestions for improvement

## 7. Difficulties

During the workshop, I faced some difficulties:

- I needed time to understand the project flow before testing.
- Some backend responses were not easy to understand at first.
- I had to check carefully because small UI or input issues could be missed.
- I needed to distinguish between frontend issues, backend issues, and integration issues.
- My testing experience was still at a basic manual testing level.

Because of these difficulties, I tested slowly and recorded observations clearly.

---

## 8. Lessons Learned

After completing this workshop, I learned more about the role of QA testing in a software project.

I learned that testing is important because it helps identify problems before the final review.

Through this workshop, I improved my understanding of:

- How to check application behavior from the user perspective.
- How to test valid and invalid input.
- How frontend and backend communicate.
- How to observe backend responses.
- How to record testing results clearly.
- How QA feedback can support project improvement.

The workshop helped me apply my AWS learning and QA Tester role to the final project **AWS Stock Analyzer**.

Although I did not independently deploy the whole AWS infrastructure, I contributed to the project by checking the application, testing basic functions, observing system behavior, and recording testing results.
